## Зміна 1: Дедуплікація замовлень

### Проблема
Якщо cron-job спрацював двічі (через redeploy, retry чи помилку), або якщо вручну запустити `send_daily_ads_report`, одне й те саме замовлення могло потрапити в звіт повторно.

### Рішення
Нова таблиця в Supabase зберігає `order_id` кожного замовлення, яке вже було в надісланому звіті. Перед формуванням нового звіту ID відсіюються.

### Зміни в БД
Створено таблицю `reported_ad_orders`:

```sql
CREATE TABLE reported_ad_orders (
    order_id BIGINT PRIMARY KEY,
    ad_title TEXT,
    order_date DATE,
    chat_id TEXT,
    reported_at TIMESTAMPTZ DEFAULT NOW()
);
CREATE INDEX idx_reported_ad_orders_date ON reported_ad_orders(order_date);
ALTER TABLE reported_ad_orders ENABLE ROW LEVEL SECURITY;
```

### Зміни в коді

**`src/database/supabase_client.py`**:
```python
def get_reported_order_ids() -> set
def mark_orders_reported(rows: list)
```

**`src/analyzer/ad_analytics.py`**:
- `build_ad_report(exclude_reported=True)` — при `True` фільтрує замовлення з `reported_ad_orders`
- `mark_report_as_sent(report)` — записує новий батч

**`src/telegram_bot/ads_bot.py`** (`send_daily_ads_report`):
- Використовує `exclude_reported=True`
- Після успішного надсилання викликає `mark_report_as_sent()`
- Якщо всі замовлення — дублі (total = 0) → **не надсилає** звіт (без спаму)

**Команди `/ads`, `/ads_today`, `/ads YYYY-MM-DD` не дедуплікують** — показують повну картину для ручного перегляду.

## Зміна 2: Виправлення атрибуції реклами

### Проблема (виявлена після ранкового звіту 11.06)

На скрінах від керівника три клієнти, які прийшли з нової реклами:

| Клієнт | Чат | Реклама на скріні |
|---|---|---|
| Іра Дмитрів | 06.06 08:36 | 😍Крута новиночка 25.05 мода+життя |
| Кристина Яровая | 05.06 14:09 | 🔥ТОП-3 креми з SPF 30 30.03.2026 |
| Мелентьева Татьяна | 06.06 09:39 | 😍Крута новиночка 25.05 мода+життя |

Усі троє оформили замовлення **10.06**, але в ранковому звіті 11.06 ці рекламні об'яви були атрибутовані **неправильно**:

| Клієнт | Звіт показав | Правильно мало бути |
|---|---|---|
| Дмитрів | Крем з вітаміном К 16.02 | Крута новиночка 25.05 |
| Яровая | ВB крем SPF 04.03 | ТОП-3 креми з SPF 30.03 |
| Мелентьева | Пакування 26.09.25 | Крута новиночка 25.05 |

### Корінь проблеми

**Sitniks тримає один чат на клієнта** (`userId` Instagram). Коли клієнт повертається через **нову** рекламу, в той самий чат додається повідомлення з новим `adInfo`. Замовлення прив'язується до цього чату через `order.chatId`.

Старий код у `get_ad_info_for_chat()` повертав **найперше** `adInfo` у чаті (найстарішу рекламу — з якої клієнт прийшов найперший раз). Це втрачало реальну атрибуцію останнього звернення.

**Приклад чату Мелентьевої**:
```
2025-10-28  adInfo: Пакування 26.09.25 (дубль)   ← старий код брав це
2025-10-28  adInfo: Пакування 26.09.25 (дубль)
2026-06-10  adInfo: 😍Крута новиночка 25.05      ← треба брати це (замовлення 10.06)
2026-06-10  adInfo: 😍Крута новиночка 25.05
2026-06-10  adInfo: 😍Крута новиночка 25.05
```

### Рішення

`get_ad_info_for_chat(chat_id, before_iso)` тепер:
1. Тягне **всі** повідомлення чату (через `get_chat_messages` з пагінацією)
2. Збирає всі повідомлення з `adInfo`
3. Якщо передано `before_iso` (= `order.createdAt`) — повертає **найсвіжіший** `adInfo` зі створенням ≤ `before_iso`
4. Fallback — найсвіжіший adInfo взагалі (якщо нема в межах вікна)

У `ad_analytics.py` тепер передаємо `before_iso=order.createdAt` для кожного замовлення.

### Перевірка

Запустили `build_ad_report` за 10.06 з новою логікою:

```
✅ Дмитрів Ірина Володимирівна → 😍Крута новиночка 25.05 мода+життя
✅ Яровая Кристина → 🔥ТОП-3 креми з SPF захистом 30 30.03.2026
✅ Мелентьева тетяна → 😍Крута новиночка 25.05 мода+життя
```
